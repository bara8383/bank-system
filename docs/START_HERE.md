# 金融システム学習プロジェクトの始め方

このドキュメントは、AI駆動開発で金融関係のシステムを再現しながら、金融システムの業務・設計・実装を学ぶための進め方を提案するものです。

## まず目指すもの

最初から本物の銀行システム全体を作ろうとせず、学習しやすい小さな銀行システムを段階的に育てます。

最初のゴールは、次の流れを安全に扱えるミニバンキングシステムです。

1. 顧客を登録する
2. 顧客に口座を作る
3. 口座へ入金する
4. 口座から出金する
5. 口座間で振込する
6. 残高と取引履歴を確認する
7. 操作の監査ログを残す

## 学習の進め方

### Phase 1: 業務の言葉を整理する

最初にコードではなく、金融システムで扱う概念を整理します。

やること:

- `docs/domain-model.md` を作り、主要な業務用語を定義する
- 顧客、口座、取引、残高、振込、監査ログの意味を整理する
- それぞれの概念が持つべき属性を書き出す

学ぶこと:

- 金融システムでは、曖昧な用語がバグや設計ミスにつながること
- 「残高」と「取引履歴」は別の概念であること
- 「顧客」と「口座」は一対一とは限らないこと

### Phase 2: 最小機能を決める

次に、最初に作る最小機能を決めます。

やること:

- `docs/mvp.md` を作り、最初に実装する範囲を決める
- 入金、出金、振込、残高照会、取引履歴照会をMVP候補にする
- 今は作らない機能も明記する

最初は作らなくてよいもの:

- 実際の外部銀行接続
- クレジットカード決済
- 本人確認書類アップロード
- 複雑な金利計算
- 株式、投資信託、FXなどの投資商品
- 高度な不正検知AI

学ぶこと:

- 金融システムは範囲が広いため、作るものと作らないものを分ける重要性
- 小さい機能でも、整合性・監査性・セキュリティを考える必要があること

### Phase 3: 重要な設計原則を決める

金融システムでは、単に動くだけでなく、間違ったお金の動きを防ぐことが重要です。

やること:

- `docs/design-principles.md` を作る
- 以下の設計原則をプロジェクトの前提として書く

設計原則の候補:

- 残高はマイナスにしない
- すべての残高変更には取引履歴を残す
- すべての重要操作には監査ログを残す
- 同じ振込依頼が二重実行されないようにする
- お金の移動は途中で片方だけ成功しないようにする
- 金額は浮動小数点では扱わない
- 認証済みユーザーだけが操作できるようにする

学ぶこと:

- トランザクション
- 整合性
- 冪等性
- 監査ログ
- 金額計算におけるデータ型の重要性

### Phase 4: 最初のデータモデルを作る

業務用語とMVPが決まったら、データモデルを考えます。

やること:

- `docs/data-model.md` を作る
- 最初のテーブル候補を整理する

テーブル候補:

- `users`: ログインユーザー
- `customers`: 顧客情報
- `accounts`: 口座
- `transactions`: 入出金や振込の取引履歴
- `transfer_requests`: 振込依頼
- `audit_logs`: 監査ログ

学ぶこと:

- 業務概念をデータ構造に落とし込む方法
- 残高をどこに持つかという設計判断
- 取引履歴を改ざんしにくくする考え方

### Phase 5: APIまたは画面の利用シナリオを書く

実装前に、利用者がどのように操作するかを文章で書きます。

やること:

- `docs/use-cases.md` を作る
- 正常系と異常系を分けて書く

ユースケース例:

- 顧客が自分の残高を確認する
- 顧客が自分の口座へ入金する
- 顧客が出金する
- 顧客が別口座へ振込する
- 残高不足の振込が拒否される
- 同じ振込依頼を再送しても二重送金されない
- 管理者が監査ログを確認する

学ぶこと:

- 正常系だけでなく異常系を考える重要性
- 金融システムでは「失敗時の扱い」が特に重要であること

### Phase 6: 実装を小さく始める

設計ドキュメントを作った後、最小機能から実装します。

おすすめの実装順:

1. プロジェクトの技術スタックを決める
2. ユーザー認証の土台を作る
3. 口座作成機能を作る
4. 入金機能を作る
5. 出金機能を作る
6. 振込機能を作る
7. 取引履歴照会を作る
8. 監査ログを追加する
9. テストを増やす
10. 異常系を強化する

## 最初の一歩

まずは `docs/domain-model.md` から作るのがおすすめです。

理由は、金融システムでは用語の理解がそのまま設計品質に影響するためです。たとえば、顧客、ユーザー、口座、残高、取引、振込依頼を曖昧にしたまま実装を始めると、後から設計を直す範囲が大きくなります。

最初の作業として、以下の問いに答えるところから始めると進めやすいです。

- このシステムの利用者は誰か
- 顧客とログインユーザーは同じものか
- 1人の顧客は複数の口座を持てるか
- 口座にはどのような状態があるか
- 入金、出金、振込はすべて同じ取引として扱えるか
- 残高はどのタイミングで変わるか
- 操作ログと取引履歴は何が違うか

## AIに依頼するときの進め方

AI駆動開発では、いきなり「銀行システムを作って」と依頼するよりも、次のように小さく依頼すると学習しやすくなります。

依頼例:

- `docs/domain-model.md に、ミニバンキングシステムの業務用語集を作ってください`
- `docs/mvp.md に、最初に作る機能と作らない機能を整理してください`
- `docs/data-model.md に、顧客、口座、取引履歴の初期データモデル案を書いてください`
- `入金処理の正常系と異常系を use-cases.md に追加してください`
- `振込処理で二重送金を防ぐ設計を提案してください`

## 注意点

このプロジェクトは学習目的のシステムです。実際の金融機関で使える本番システムをそのまま作るものではありません。

ただし、学習目的であっても、金融システムらしい設計観点は大切にします。特に、お金の整合性、監査ログ、認証・認可、テスト、異常系の扱いは早い段階から意識します。
