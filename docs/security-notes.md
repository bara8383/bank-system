# ミニバンキングシステム セキュリティメモ

このドキュメントは、MVP段階から意識するセキュリティ上の注意点を整理します。学習用であっても、金融システムらしい認証、認可、監査、秘密情報の扱いを前提にします。

## 認証

- パスワードは平文で保存しない。
- パスワードは安全なハッシュ方式で保存する。
- ログイン失敗の詳細理由を利用者へ出しすぎない。
- セッションやトークンには有効期限を設ける。
- 重要操作では将来的に多要素認証を検討する。

## 認可

- 顧客ユーザーは自分に紐づく口座だけ操作できる。
- 管理者と運用担当者の権限を分ける。
- 口座IDを指定された場合でも、必ず所有者または権限を確認する。
- 監査ログ照会は限定されたロールだけに許可する。
- MVPでは監査ログ照会を `admin` のみに限定し、`operator` ロールはMVP対象外とする。

## 監査

- 重要操作の成功と失敗を監査ログに残す。
- 監査ログには `actor_user_id`、`action_type`、`target_type`、`target_id`、`result`、`failure_reason`、`occurred_at`、`ip_address`、`user_agent` を含める。
- `failure_reason` は利用者向け詳細ではなく、運用調査に必要な安全な分類または短い理由にする。
- domain error に由来する `failure_reason` は、`internal/domain` の safe failure category helper が返す固定分類値を使う。helper は raw request body、password、token、secret、CSRF token、セッションID、未加工の自由入力値を保存・返却する用途ではない。
- リクエスト同一性の調査が必要な場合は、raw request body ではなく request body hash を保存する設計案にする。
- 残高変更の理由は取引履歴に残す。
- 監査ログと取引履歴は削除せず、追記型を基本にする。

## 秘密情報と個人情報

- パスワード、トークン、秘密鍵はログに出力しない。
- 監査ログにはパスワード、token、secret、CSRF token、セッションID、raw request body、過剰な個人情報を保存しない。
- 個人情報は必要最小限だけ保存する。
- 開発用のダミーデータに実在の個人情報を使わない。
- 環境変数やシークレット管理の仕組みを使い、認証情報をリポジトリにコミットしない。

## 入力検証

- 金額は正の整数のみ受け付ける。
- 口座番号やログインIDの形式を検証する。
- 検索条件には上限や期間制限を設ける。
- ユーザー入力をSQLやログへ直接埋め込まない。

## 今後検討する対策

- 多要素認証
- レート制限
- 不審な振込パターンの検知
- 監査ログの改ざん検知
- データベース暗号化
- バックアップと復旧手順
