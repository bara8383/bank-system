# ミニバンキングシステム 設計原則

このドキュメントは、学習用ミニバンキングシステムを設計・実装するときに守る前提を定義します。金融システムでは「動くこと」だけでなく、「間違ったお金の動きを防ぐこと」と「後から説明できること」が重要です。

## 基本原則

### 1. 残高をマイナスにしない

出金や振込では、処理前に利用可能残高を確認します。残高不足の場合は処理を拒否し、残高を更新しません。

### 2. 金額は浮動小数点で扱わない

金額は整数の最小通貨単位で扱います。MVPではJPYのみを対象にするため、1円を1として保存します。

### 3. すべての残高変更に取引履歴を残す

入金、出金、振込出金、振込入金など、残高を変える処理では必ず取引履歴を作成します。履歴は原則として削除せず、取り消しが必要な場合は取消取引を追加します。

### 4. 重要操作に監査ログを残す

ログイン、顧客登録、口座作成、入金、出金、振込、権限変更などは監査ログに記録します。監査ログには実行者、対象、操作種別、実行日時、結果、失敗理由を残します。

### 5. 認証済みユーザーだけが操作できる

口座、残高、取引履歴、振込依頼などの操作は認証を前提にします。未認証ユーザーは業務データへアクセスできません。

### 6. 認可を必ず確認する

認証済みであっても、すべての口座を操作できるわけではありません。顧客ユーザーは自分に紐づく口座のみ操作でき、管理者や運用担当者の操作はロールに応じて制限します。

### 7. お金の移動は原子的に扱う

振込では、振込元口座の残高減少、振込先口座の残高増加、取引履歴作成、振込依頼更新をひとまとまりとして扱います。途中で片方だけ成功しないよう、データベーストランザクションを使用します。

残高変更に成功した場合、口座残高の更新と取引履歴の作成は同じデータベーストランザクションに含めます。残高変更に成功したのに取引履歴がない状態、または取引履歴だけがあり残高が変わらない状態は禁止します。

MVPでは金額は正の整数、通貨はJPYのみとし、残高は0未満にしません。

成功時の最小トランザクション境界は次のとおりです。MVP初期案では、残高変更を伴う成功操作の成功監査ログも同じPostgreSQLデータベーストランザクションに含めます。

| 操作 | 同じデータベーストランザクションに含める更新 |
| --- | --- |
| 入金 | 対象口座の残高増加、入金取引履歴の作成、入金成功監査ログの作成 |
| 出金 | 対象口座の残高減少、出金取引履歴の作成、出金成功監査ログの作成 |
| 振込 | 振込元口座の残高減少、振込先口座の残高増加、振込出金取引履歴の作成、振込入金取引履歴の作成、振込依頼の成功状態更新、振込成功監査ログの作成 |

監査ログの記録要否と保存境界は、次のように分けます。

| ケース | 業務データ更新 | 監査ログの保存境界 |
| --- | --- | --- |
| 残高変更を伴う成功操作 | 口座残高、取引履歴、振込依頼状態などを更新する | 成功監査ログを業務データ更新と同じPostgreSQLデータベーストランザクションに含める |
| 権限不足、残高不足、入力不備、存在しない対象などの業務拒否 | 業務データを更新しない | 失敗監査ログを独立したデータベーストランザクションで残す |
| データベーストランザクション途中失敗 | 業務データ更新をロールバックする | ロールバック後に失敗監査ログを独立したデータベーストランザクションで残す |
| 監査ログ書き込み失敗 | 対象業務処理を成功扱いにしない | 成功監査ログが書けない残高変更や権限変更はMVPではfail closedとする |

失敗時監査ログは不要ではなく、業務データ更新のロールバックと独立して残す必要があります。一方で、監査ログ自体の書き込み失敗をどう運用通知、再送、補償するかは将来検討とします。並行更新制御方式も未確定であり、別の設計判断として扱います。

### 8. 二重実行を防ぐ

通信リトライや画面の二重送信により、同じ振込依頼が複数回処理されないようにします。振込依頼には冪等性キーを持たせ、同じキーの処理結果を再利用します。

### 9. 状態遷移を明示する

口座や振込依頼は状態を持ちます。状態ごとに許可される操作を明確にし、停止中口座への出金や成功済み振込依頼の再実行などを防ぎます。

### 10. エラーは調査可能にする

異常系では、利用者向けの安全なエラーメッセージと、運用調査向けの詳細なログを分けます。秘密情報やパスワードはログに出力しません。

## 実装時のチェックリスト

- 金額の型は整数になっているか。
- 残高更新と取引履歴作成が同じトランザクションに入っているか。
- 振込元と振込先の更新が同じトランザクションに入っているか。
- 残高不足時に残高が変わらないか。
- 重要操作の成功・失敗が監査ログに残るか。
- 認証と認可を両方確認しているか。
- 冪等性キーの一意制約があるか。
- 失敗時に機密情報をログへ出していないか。
