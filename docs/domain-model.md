# ミニバンキングシステム 業務用語集

このドキュメントは、ミニバンキングシステムで使う主要な業務用語を定義するものです。実装時に用語の意味がぶれないようにし、データモデル・API・画面・テストで同じ言葉を使うことを目的とします。

## 用語一覧

| 用語 | 英語名 / 候補名 | 意味 | 主な属性例 | 補足・設計メモ |
| --- | --- | --- | --- | --- |
| 顧客 | Customer | 銀行サービスを利用する個人または法人。口座の所有者になり得る主体。 | 顧客ID、氏名、メールアドレス、電話番号、住所、ステータス、登録日時 | 顧客と口座は一対一とは限らない。1人の顧客が複数口座を持つことがある。 |
| ログインユーザー | User | システムへログインして操作する利用者。顧客本人、管理者、運用担当者などを含む。 | ユーザーID、ログインID、パスワードハッシュ、ロール、有効/無効、最終ログイン日時 | 顧客情報とは分けて考える。管理者は顧客ではない場合がある。 |
| 口座 | Account | 顧客が入金・出金・振込に利用するお金の保管単位。 | 口座ID、口座番号、顧客ID、口座種別、通貨、ステータス、開設日時 | 残高や取引履歴と関連する中心的な概念。 |
| 口座番号 | Account Number | 顧客や外部利用者が口座を識別するための番号。 | 口座番号、発行日時、利用状態 | 内部処理では連番IDなどの口座IDを使い、口座番号は表示・検索用として扱うとよい。 |
| 口座種別 | Account Type | 普通預金、当座預金など、口座の種類を表す区分。 | 種別コード、種別名 | ミニバンキングシステムでは最初は普通預金のみでもよい。 |
| 口座ステータス | Account Status | 口座が利用可能かどうかを示す状態。 | 有効、停止中、解約済み | 停止中や解約済みの口座では入出金・振込を制限する。 |
| 残高 | Balance | ある時点で口座に存在する利用可能な金額。 | 口座ID、金額、通貨、最終更新日時 | 取引履歴とは別概念。残高は最新状態、取引履歴は変化の記録。 |
| 利用可能残高 | Available Balance | 出金や振込に利用できる金額。 | 口座ID、金額、通貨 | 将来、保留中取引や凍結金額を扱う場合、単純な残高と分ける。MVPでは残高と同じでもよい。 |
| 金額 | Amount | 入金・出金・振込などで扱うお金の量。 | 数値、通貨 | 浮動小数点では扱わず、整数の最小通貨単位などで扱う。例: 1000円は1000として保存。 |
| 通貨 | Currency | 金額の単位。 | 通貨コード、通貨名、小数桁 | MVPではJPYのみを前提にしてよい。将来の多通貨対応では必須。 |
| 取引 | Transaction | 口座残高を変化させる業務上の出来事。入金、出金、振込の引落・入金などを含む。 | 取引ID、口座ID、取引種別、金額、取引日時、残高変更後金額、説明、関連取引ID | すべての残高変更には取引を残す。削除せず、必要なら取消取引を追加する。 |
| 取引履歴 | Transaction History | 口座に対して発生した取引の時系列記録。 | 口座ID、取引一覧、検索期間 | 顧客への明細表示や監査の基礎になる。 |
| 取引種別 | Transaction Type | 取引の種類を表す区分。 | 入金、出金、振込出金、振込入金、取消 | 残高の増減方向を判断するためにも使う。 |
| 入金 | Deposit | 口座の残高を増やす操作または取引。 | 入金ID、口座ID、金額、受付日時、実行日時、実行者 | MVPでは外部現金や外部決済との接続は行わず、システム内の操作として扱う。 |
| 出金 | Withdrawal | 口座の残高を減らす操作または取引。 | 出金ID、口座ID、金額、受付日時、実行日時、実行者 | 残高不足の場合は拒否する。残高をマイナスにしない。 |
| 振込 | Transfer | ある口座から別の口座へ資金を移動する業務。 | 振込ID、振込元口座ID、振込先口座ID、金額、ステータス、受付日時、実行日時 | 振込元の出金と振込先の入金は一体として扱い、片方だけ成功しないようにする。 |
| 振込依頼 | Transfer Request | 利用者から受け付けた振込の依頼内容。 | 依頼ID、依頼者ID、振込元口座ID、振込先口座番号、金額、依頼日時、冪等性キー、ステータス | 同じ依頼が再送されても二重送金しないよう、冪等性キーを持たせる。 |
| 振込ステータス | Transfer Status | 振込依頼または振込処理の進行状態。 | 受付済み、処理中、成功、失敗、取消済み | 失敗理由を残すと調査しやすい。 |
| 冪等性キー | Idempotency Key | 同じ操作の重複実行を防ぐための識別子。 | キー、対象操作、作成日時、有効期限 | 通信リトライや画面の二重送信による二重入金・二重送金を防ぐ。 |
| 監査ログ | Audit Log | 誰が、いつ、どの重要操作を行ったかを追跡するための記録。 | 監査ログID、実行者ID、操作種別、対象ID、実行日時、IPアドレス、結果 | 残高変更だけでなく、ログイン、口座作成、権限変更なども記録対象にする。 |
| 操作種別 | Action Type | 監査ログに記録する操作の種類。 | ログイン、顧客登録、口座作成、入金、出金、振込、照会、権限変更 | 重要操作を分類し、検索・調査しやすくする。 |
| ロール | Role | ログインユーザーが持つ権限の種類。 | 顧客、管理者、運用担当者 | 顧客は自分の情報だけを見られるようにし、管理者操作は監査ログを必ず残す。 |
| 認証 | Authentication | 利用者が本人であることを確認する仕組み。 | ログインID、パスワード、セッション、トークン | 認証されていないユーザーには残高照会や振込を許可しない。 |
| 認可 | Authorization | 認証済みユーザーが、その操作を実行できるか判断する仕組み。 | ユーザーID、ロール、対象リソース、許可/拒否 | 例: 顧客Aが顧客Bの口座を照会できないようにする。 |
| トランザクション | Database Transaction | 複数のデータ更新をひとまとまりとして成功または失敗させる仕組み。 | 開始時刻、コミット、ロールバック | 振込では、振込元の残高減少と振込先の残高増加を同じトランザクションで扱う。 |
| 整合性 | Consistency | データが業務ルールに矛盾しない状態。 | ルール、検証結果 | 例: 残高がマイナスにならない、取引履歴の合計と残高が矛盾しない。 |

## 主要な関係

- 1人の顧客は、0個以上の口座を持つことができます。
- 1つの口座は、1人以上の顧客に紐づく可能性があります。ただし、MVPでは1口座につき1顧客として簡略化してもよいです。
- 1つの口座には、0件以上の取引が発生します。
- 1件の振込は、振込元口座の出金取引と振込先口座の入金取引を発生させます。
- 重要操作は、業務データの更新とは別に監査ログへ記録します。

## MVPで優先して扱う用語

最初の実装では、次の用語を中心に設計します。

1. 顧客
2. ログインユーザー
3. 口座
4. 残高
5. 取引
6. 入金
7. 出金
8. 振込
9. 振込依頼
10. 監査ログ

## MVPではまだ簡略化する用語

次の概念は重要ですが、最初は単純化して扱います。

- 口座種別: 普通預金のみとする。
- 通貨: JPYのみとする。
- 利用可能残高: 残高と同じ値として扱う。
- 共同名義口座: 扱わず、1口座につき1顧客とする。
- 外部銀行接続: 扱わず、システム内部の口座間振込のみとする。
