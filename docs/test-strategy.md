# ミニバンキングシステム テスト戦略

このドキュメントは、MVPで何をどのようにテストするかを整理します。金融システムでは正常系だけでなく、残高不足、二重実行、権限不足、途中失敗などの異常系テストが特に重要です。

## テスト方針

- 金額計算と残高更新を最優先でテストする。
- 取引履歴と監査ログが必ず作られることを確認する。
- 正常系よりも異常系を意識してテストケースを増やす。
- 振込処理では原子性と冪等性を重点的に確認する。
- 認証・認可を回避できないことを確認する。

## 単体テスト

| 対象 | 確認内容 |
| --- | --- |
| 金額バリデーション | 0円以下、小数、通貨不一致を拒否する。 |
| 残高計算 | 入金で増え、出金で減り、残高不足で変わらない。 |
| 状態判定 | 停止中・解約済み口座の操作を拒否する。 |
| 権限判定 | 他人の口座操作を拒否する。 |
| 冪等性判定 | 同じ冪等性キーの再実行を防ぐ。 |

## 結合テスト

| シナリオ | 確認内容 |
| --- | --- |
| 入金成功 | 残高、取引履歴、成功監査ログが同じ成功結果として作られる。 |
| 出金成功 | 残高が減り、取引履歴と成功監査ログが作られる。 |
| 出金失敗 | 残高不足時に残高が変わらず、失敗監査ログが残る。 |
| 振込成功 | 振込元と振込先の残高、2件の取引履歴、振込依頼状態、成功監査ログが整合する。 |
| 振込失敗 | 途中失敗時に両口座の残高が変わらない。 |
| 二重送信 | 同じ振込依頼を再送しても1回分だけ処理される。 |

## セキュリティテスト

- 未認証ユーザーが残高照会できないこと。
- 顧客Aが顧客Bの口座を照会・操作できないこと。
- `admin` 以外が監査ログを照会できないこと。
- MVPでは `operator` を監査ログ照会の権限表・照会対象に含めないこと。
- パスワードやトークンがレスポンスやログに出ないこと。
- 不正な入力値でサーバーエラーではなく検証エラーになること。

## データ整合性テスト

- `accounts.balance_amount` が0未満にならないこと。
- 取引履歴の増減と現在残高が矛盾しないこと。
- 振込の出金取引と入金取引が関連付けられること。
- 失敗した振込依頼が成功取引を持たないこと。
- `transactions.balance_after` が0未満にならないこと。

## 残高変更と取引履歴のトランザクションテスト

将来、入金、出金、振込を実装するときは、口座残高の更新と取引履歴の作成が同じデータベーストランザクションで整合することを確認します。

| シナリオ | 確認内容 |
| --- | --- |
| 入金成功 | 残高増加、`deposit` 取引履歴、入金成功監査ログが同じデータベーストランザクションで作られ、`balance_after` が更新後残高と一致する。 |
| 出金成功 | 残高減少、`withdrawal` 取引履歴、出金成功監査ログが同じデータベーストランザクションで作られ、`balance_after` が更新後残高と一致する。 |
| 出金失敗 | 残高不足時に残高も取引履歴も変わらず、失敗監査ログが独立して残る。 |
| 振込成功 | 2口座の残高、`transfer_debit` と `transfer_credit` の2件の取引履歴、振込依頼状態、振込成功監査ログが同じデータベーストランザクションで整合する。 |
| 振込失敗 | 片方の残高だけ、または片方の取引履歴だけが残らず、失敗監査ログが独立して残る。 |

## 監査ログテスト

| シナリオ | 確認内容 |
| --- | --- |
| 重要操作成功 | 入金、出金、振込の成功時に、業務データ、取引履歴、成功監査ログが整合して作られる。 |
| 業務拒否 | 残高不足、権限不足、不正金額、存在しない対象では、業務データが変わらず失敗監査ログが残る。 |
| 途中失敗rollback | 残高更新後・取引履歴作成前、振込元更新後・振込先更新前、取引履歴作成後・commit前などの疑似エラーで業務データがrollbackされ、rollback後に失敗監査ログが独立して残る。 |
| 監査ログ書き込み失敗 | 成功監査ログを書けない残高変更や権限変更は、MVPでは対象業務処理が成功扱いにならない。 |
| マスキング | 監査ログに password、token、secret、CSRF token、セッションID、raw request body、過剰な個人情報が含まれない。 |
| 閲覧権限 | `admin` 以外が監査ログを照会できず、MVPでは `operator` を権限表・照会対象に含めない。 |

## DB transaction 途中失敗のrollbackテスト

業務拒否とは別に、実装上の途中失敗を注入し、同じデータベーストランザクションに含めるべき更新が一部だけ残らないことを確認します。

| シナリオ | 確認内容 |
| --- | --- |
| 入金途中失敗 | 残高更新後・取引履歴作成前の疑似エラーで、残高、取引履歴、成功監査ログが残らず、失敗監査ログだけが独立して残る。 |
| 出金途中失敗 | 残高減少後・取引履歴作成前の疑似エラーで、残高、取引履歴、成功監査ログが残らず、失敗監査ログだけが独立して残る。 |
| 振込途中失敗 | 振込元更新後・振込先更新前、片側取引履歴作成後、またはcommit前の疑似エラーで、2口座残高、2取引履歴、振込依頼状態、成功監査ログがすべてrollbackされ、失敗監査ログだけが独立して残る。 |

## 手動確認

MVP初期では、以下を手動でも確認します。

1. 顧客登録から口座作成までの流れ。
2. 入金後に残高と取引履歴が更新されること。
3. 出金後に残高と取引履歴が更新されること。
4. 残高不足の出金が拒否されること。
5. 振込後に両口座の残高と取引履歴が更新されること。
6. 管理者が監査ログを確認できること。
