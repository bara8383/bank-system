## Finding 1: 認証方式・セッション/トークン管理の仕様が未確定

- 重大度: High
- 人間確認要否: あり

### 根拠

`docs/security-notes.md` では、パスワードを平文保存しないこと、安全なハッシュ方式、有効期限付きセッション/トークン、将来的な多要素認証が挙げられている。一方で、MVP実装前に必要な以下の具体仕様は未定義である。

- パスワードハッシュ方式（例: Argon2id / bcrypt など）とパラメータ
- セッション方式（Cookie ベースか Bearer token か）
- セッション/トークンの有効期限、更新、失効、ログアウト時の扱い
- CSRF 対策の要否（Cookie ベースの場合）
- ログイン失敗時のレート制限・アカウントロック方針

### 影響

認証基盤を曖昧なまま実装すると、パスワードハッシュ強度不足、長寿命トークンの漏えい悪用、ログアウト後も有効なトークン、ブルートフォース攻撃への耐性不足が発生しやすい。金融学習プロジェクトでは、残高照会・振込・監査ログなどの重要操作すべてが認証に依存するため、後からの修正範囲も大きくなる。

### 推奨修正

次サイクルで、実装前提として `docs/security-notes.md` または新規の設計ログに以下を小さく決める。

1. MVPで採用する認証方式。
2. パスワードハッシュ方式と最低パラメータ。
3. セッション/トークンの保存場所、有効期限、失効方法。
4. ログイン失敗時のレスポンス方針、試行回数制限、監査ログ記録。
5. Cookie を使う場合の Secure / HttpOnly / SameSite と CSRF 方針。

### 次サイクル planner への入力

- accepted scope 候補: 「MVP認証・セッション管理方針を文書化し、実装時チェックリストへ反映する」
- 実装する場合は、認証の最小APIより先にセキュリティ仕様を固定する。

## Finding 2: 認可モデルの粒度と管理者/運用担当者の権限分離が未確定

- 重大度: High
- 人間確認要否: あり

### 根拠

設計原則では、顧客ユーザーは自分に紐づく口座のみ操作でき、管理者や運用担当者の操作はロールに応じて制限するとしている。MVPでも、残高照会、取引履歴照会、監査ログ照会、入出金・振込に認可を求めている。しかし、ロールごとの許可操作、管理者が顧客口座へ入出金できるか、運用担当者が監査ログだけ閲覧できるか、といった権限表はまだ存在しない。

### 影響

ロール定義が曖昧だと、実装時にエンドポイント単位で個別判断が入り、権限昇格・水平権限不備（顧客Aが顧客Bの口座を見る）・管理者権限の過大付与が起きやすい。特に金融系では、入出金・振込・監査ログ閲覧・権限変更を同一管理者ロールにまとめると、内部不正や誤操作の影響が大きくなる。

### 推奨修正

次サイクルで、MVPの簡易RBAC表を追加する。

| ロール | 許可候補 | 禁止候補 | 監査ログ |
| --- | --- | --- | --- |
| customer | 自分の残高/履歴照会、自分の口座からの出金/振込 | 他顧客口座、監査ログ照会、権限変更 | 重要操作を記録 |
| admin | 顧客登録、口座作成、監査ログ照会 | 自己承認なしの権限変更などは要検討 | 全操作を記録 |
| operator | 運用調査用の限定照会 | 残高変更、権限変更 | 全操作を記録 |

あわせて、API設計時には「URLやリクエストの `account_id` を信用せず、必ずDB上の所有関係で検証する」ことを明記する。

### 次サイクル planner への入力

- accepted scope 候補: 「MVPロール/権限マトリクスを作成し、認可チェックの実装単位を定義する」
- 人間確認事項: 管理者が入金・出金・振込を代行できるか、運用担当者ロールをMVPに含めるか。

## Finding 3: 監査ログは定義されているが、改ざん耐性・失敗時記録・機微情報マスキングの具体性が不足

- 重大度: Medium
- 人間確認要否: 一部あり

### 根拠

既存文書では、重要操作の成功/失敗を監査ログに残し、実行者、対象、操作種別、結果、日時を含めること、ログやレスポンスにパスワード・トークンを出さないことが示されている。一方で、以下は未確定である。

- 監査ログ書き込みに失敗した場合、業務処理を失敗させるかどうか
- 監査ログの保存期間、削除禁止・論理削除・閲覧権限
- 改ざん検知（ハッシュチェーン、署名、追記専用権限など）のMVPでの扱い
- IPアドレスやUser-Agent、失敗理由に含める情報の粒度
- ログ出力時の個人情報・口座番号・トークン等のマスキング規則

### 影響

監査ログは事故調査と不正抑止の中核だが、業務処理と監査ログの整合性が曖昧だと「残高は変わったが監査ログがない」「失敗時の攻撃痕跡が残らない」「ログに秘密情報を残して二次漏えいする」などのリスクがある。

### 推奨修正

MVPでは過剰な仕組みにせず、最低限として以下を明文化する。

1. 残高変更・権限変更・ログインは、成功/失敗とも監査ログ対象。
2. 残高変更系は、業務更新と監査ログ記録を同一DBトランザクションに含めるか、少なくとも失敗時の扱いを設計判断として記録する。
3. 監査ログの本文にはパスワード、トークン、秘密鍵、生の認証ヘッダーを保存しない。
4. 顧客向けエラーと運用向け失敗理由を分離し、運用向けにも機微情報を入れすぎない。
5. 改ざん検知はMVP後でもよいが、追記型・削除禁止をDB権限/アプリ権限で支える方針を置く。

### 次サイクル planner への入力

- accepted scope 候補: 「監査ログの記録対象、失敗時扱い、マスキング規則をMVP設計に追加する」
- 人間確認事項: MVPで監査ログ書き込み失敗時に業務処理を止めるか。

## Finding 4: 振込・入出金の並行実行対策が設計文書上まだ薄い

- 重大度: High
- 人間確認要否: なし（実装設計で具体化が必要）

### 根拠

設計原則では、振込元の残高減少、振込先の残高増加、取引履歴作成、振込依頼更新をデータベーストランザクションで一体化するとしている。また、冪等性キーで二重送金を防ぐ方針がある。しかし、PostgreSQLでの行ロック、分離レベル、更新順序、残高不足判定と更新をどのSQLで一体化するかは未定義である。

### 影響

単純に「残高を読み、アプリで判定し、後で更新」する実装になると、同時出金・同時振込で残高不足をすり抜ける可能性がある。振込元/振込先のロック順序が不定だとデッドロックも起きやすい。冪等性キーだけでは、別キーでの並行操作による過剰出金は防げない。

### 推奨修正

実装前に、残高変更のDBトランザクション方針を文書化する。

- 出金/振込元口座は `SELECT ... FOR UPDATE` などで行ロックする、または `UPDATE accounts SET balance_amount = balance_amount - ? WHERE id = ? AND balance_amount >= ?` のように条件付き更新で不足を原子的に判定する。
- 振込では複数口座ロックの順序を口座ID昇順などに固定する。
- `accounts.balance_amount >= 0` のDB制約を必須とする。
- 取引履歴と残高更新を同一トランザクションでコミットする。
- 並行実行テストをMVP完了条件または次段階テストとして追加する。

### 次サイクル planner への入力

- accepted scope 候補: 「PostgreSQL前提の残高更新・振込トランザクション方針を設計文書へ追加する」

## Finding 5: 冪等性キーのスコープ・保存期間・リクエスト同一性検証が未確定

- 重大度: Medium
- 人間確認要否: あり

### 根拠

データモデル案では `transfer_requests.idempotency_key` を依頼者または振込元口座の範囲で一意にする案がある。ユースケースでも同じ冪等性キーの成功済み依頼がある場合は既存結果を返すとしている。一方で、同一キーで異なる金額・宛先が送られた場合の扱い、キーの期限、失敗済み/処理中ステータスの再送、キーの推測困難性は未定義である。

### 影響

同じキーで異なるリクエストを受け入れると、利用者・攻撃者・バグにより意図しない振込結果の再利用や上書きが起きる。処理中の再送を誤って二重処理すると二重送金になり、失敗済み再送を常に拒否すると利用者回復性が下がる。キーのスコープが広すぎても狭すぎても、衝突や重複防止漏れにつながる。

### 推奨修正

MVPの冪等性仕様として以下を決める。

1. 一意制約は `requester_user_id + source_account_id + idempotency_key` など、操作主体と対象を含める。
2. 同一キーで、振込先・金額・通貨・振込元が異なる場合は409相当の競合として拒否する。
3. `processing` の再送時は二重実行せず、状態確認または安全な再試行方針を返す。
4. キーの有効期限/保存期間を決める。
5. キーはクライアント生成でも推測困難な値を求める。

### 次サイクル planner への入力

- accepted scope 候補: 「振込依頼の冪等性仕様を詳細化し、DB一意制約案と異常系を追加する」
- 人間確認事項: 冪等性キーの保存期間をMVPで固定するか。

## Finding 6: 個人情報・口座番号・取引履歴のデータ分類と露出制御が未定義

- 重大度: Medium
- 人間確認要否: あり

### 根拠

セキュリティメモでは個人情報を必要最小限だけ保存し、実在の個人情報を開発用ダミーデータに使わない方針がある。データモデル案には顧客の氏名、メールアドレス、電話番号、住所、口座番号、取引履歴、監査ログが含まれる。これらのデータについて、APIレスポンスでのマスキング、ログ出力禁止、検索権限、開発/テストデータ方針はまだ未定義である。

### 影響

実装時に便利さを優先して顧客情報や口座番号を丸ごとレスポンス・ログ・監査ログに含めると、最小権限・最小開示に反する。取引履歴は行動履歴でもあり、漏えい時の影響が大きい。学習用であっても、後からマスキング・データ分類を足すより、最初に扱いを分けた方が安全である。

### 推奨修正

MVP向けの簡易データ分類を追加する。

- Secret: パスワード、トークン、秘密鍵。保存時ハッシュ/暗号化、ログ禁止。
- Sensitive PII: 氏名、住所、電話番号、メールアドレス。必要なAPIだけ返す。ログは原則マスク。
- Financial Sensitive: 口座番号、残高、取引履歴。本人または限定ロールのみ閲覧。ログは最小化。
- Operational: 監査ログ、IP、User-Agent。管理者/運用担当者のみ閲覧。

### 次サイクル planner への入力

- accepted scope 候補: 「MVPのデータ分類・マスキング・レスポンス最小化方針をセキュリティ文書に追加する」

## Finding 7: API入力検証・検索制限・エラー応答の標準が未定義

- 重大度: Medium
- 人間確認要否: なし

### 根拠

既存文書では、金額は正の整数、口座番号やログインIDの形式検証、検索条件の上限や期間制限、SQLやログへの直接埋め込み禁止が示されている。ただし、API実装で共通化すべき入力制約、ページング、最大期間、エラーコード、利用者向けメッセージと内部ログの分離はまだ具体化されていない。

### 影響

入力検証が各ハンドラに分散すると、検証漏れ、サーバーエラー露出、SQLインジェクション、ログインID列挙、取引履歴の大量取得によるDoSや情報過多なエラーが起きやすい。

### 推奨修正

REST APIを実装する前に、以下の小さな標準を置く。

- 金額、口座番号、ログインID、メールアドレス、検索期間、ページサイズの制約。
- 取引履歴・監査ログ検索の最大期間、最大件数、ページング必須化。
- エラー応答形式と、認証失敗/認可失敗/存在しないリソースの情報露出方針。
- SQLは必ずプレースホルダ/クエリビルダを使う方針。
- ログ出力時のリクエストボディ丸ごと出力禁止。

### 次サイクル planner への入力

- accepted scope 候補: 「REST API入力検証・エラー応答・検索制限の標準を設計文書化する」

## Finding 8: シークレット管理・設定管理・ローカル開発時の漏えい防止が未定義

- 重大度: Low
- 人間確認要否: なし

### 根拠

セキュリティメモでは、環境変数やシークレット管理の仕組みを使い、認証情報をリポジトリにコミットしないとしている。現時点ではアプリ実装がないため直接の漏えいはないが、今後 PostgreSQL 接続情報、JWT署名鍵、Cookie署名鍵、管理者初期パスワードなどが必要になる。

### 影響

初期実装時に `.env` やdocker composeのサンプルへ実シークレットを入れると、履歴に残りやすい。学習用でも、秘密情報の扱いを早めに標準化しないと、テスト・ログ・ドキュメントに秘密情報を貼る運用が定着する。

### 推奨修正

- `.env.example` にはダミー値のみ置く方針を明記する。
- 実シークレットをコミットしないための `.gitignore` 方針を実装時に確認する。
- アプリ起動時、必須シークレットがデフォルト値のままなら起動失敗させる。
- テスト用シークレットと開発用シークレットを分ける。

### 次サイクル planner への入力

- accepted scope 候補: 「設定/シークレット管理の最小方針と `.env.example` 運用ルールを追加する」

## 全体所見

現時点のrepoは実装コードがなく、ドキュメント上は金融システムで重要な認証、認可、監査、冪等性、トランザクション、入力検証、秘密情報保護の論点がすでに挙げられている。次のリスクは、実装開始時にこれらを個別ハンドラの判断へ落としてしまい、横断仕様が不足することである。

次サイクルでは、コード実装に進む前に以下のいずれかを小さく採択すると、後続実装のセキュリティ品質が安定する。

1. 認証・セッション管理方針
2. ロール/権限マトリクス
3. 残高更新/振込トランザクション方針
4. 冪等性キー詳細仕様
5. 監査ログとデータ分類/マスキング方針
