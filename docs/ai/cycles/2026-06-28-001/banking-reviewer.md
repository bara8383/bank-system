# Banking Reviewer Review: 2026-06-28-001

## レビュー種別

- 実装差分: なし（Go ソースや DB マイグレーションは未作成）
- レビュー対象: repo 全体の既存設計ドキュメント

## Finding 1: 元帳の単位と残高算出ルールがまだ曖昧

### 根拠

`docs/data-model.md` では現在残高を `accounts.balance_amount` に保持し、残高変更根拠を `transactions` に追記する方針がある。一方で、取引履歴から現在残高を検証する方法は「将来的に」追加するとされている。`transactions.amount` は常に正の整数で、増減方向は `transaction_type` から読み取る前提に見えるが、残高検証時にどの種別を加算・減算するかの正式な対応表はまだない。

### 影響

実装時に、入金・出金・振込出金・振込入金・取消の符号解釈が処理ごとに分散すると、`accounts.balance_amount` と `transactions.balance_after` の不整合を検出しにくくなる。特に取消や将来の補正取引を追加したとき、金額は正でも残高方向が逆になるため、集計・明細表示・監査調査で誤解が生じる可能性がある。

### 推奨修正

次サイクルでは、実装前に `transaction_type` ごとの残高方向を明文化する。例として、`deposit` と `transfer_credit` は加算、`withdrawal` と `transfer_debit` は減算、`reversal` は取消対象取引の逆方向、のように定義する。あわせて、`balance_after` は各口座内の取引順序に基づく値であること、検証では「初期残高 + 取引種別別の符号付き合計 = 現在残高」を確認することを設計に含める。

### 次サイクル planner への入力

- `transaction_type` と残高増減方向の対応表を accepted scope 候補に入れる。
- `accounts.balance_amount` と `transactions.balance_after` の整合性検証方針を docs に追加するか、最初の実装タスクへ含める。

## Finding 2: 振込の冪等性スコープとペイロード不一致時の扱いが不足

### 根拠

設計原則とユースケースでは、冪等性キーにより同じ振込依頼の二重実行を防ぎ、成功済み依頼がある場合は既存結果を返す方針が示されている。データモデルでは `transfer_requests.idempotency_key` を依頼者または振込元口座の範囲で一意にする案があるが、同じキーで金額・振込先・振込元が異なる依頼が来た場合の扱いは明記されていない。

### 影響

同一キーを異なる内容に再利用した場合、既存結果を返すだけだと、利用者が意図しない振込結果を正常応答として受け取る可能性がある。逆に一意制約だけで失敗させる場合も、利用者向けエラー、監査ログ、失敗した依頼を保存するかどうかが揺れる。これは二重送金防止だけでなく、誤送金調査や障害復旧時の説明責任にも影響する。

### 推奨修正

冪等性判定は「同じスコープ、同じ冪等性キー、同じリクエスト内容なら同じ結果を返す」「同じキーだが内容が異なる場合は冪等性キー衝突として拒否する」と明文化する。保存項目には、比較対象となるリクエスト内容またはハッシュを含めることを検討する。スコープは MVP では `requested_by_user_id + idempotency_key` または `source_account_id + idempotency_key` のどちらかに固定し、人間確認事項として選定する。

### 次サイクル planner への入力

- 冪等性キーの一意スコープを 1 つに決めるタスクを候補化する。
- 同一キー・異内容リクエストを拒否し、監査ログへ記録する仕様を候補化する。

## Finding 3: 振込ステータスの状態遷移と失敗記録の境界が未定義

### 根拠

`transfer_requests.status` には `accepted`、`processing`、`succeeded`、`failed`、`cancelled` が候補としてある。ユースケースでは振込依頼を受付済みとして作成し、同一トランザクション内で残高更新・取引作成・成功更新を行う流れがある。一方、どの失敗を `transfer_requests` に `failed` として残し、どの失敗は依頼作成前の入力検証エラーとして扱うかがまだ分かれていない。

### 影響

残高不足、振込先不存在、権限不足、口座停止、DB障害などがすべて同じ「失敗」として扱われると、残すべき監査証跡と残してはいけない依頼データの境界が曖昧になる。特に `processing` 中に障害が起きた場合、再実行してよいのか、人手調査が必要なのかを判定しづらくなる。

### 推奨修正

振込依頼の状態遷移表を作成し、許可される遷移を限定する。例として、入力・認可・口座存在の検証に通ったものだけを `accepted` として保存し、残高不足など業務拒否は `failed` として保存するかどうかを決める。DB障害やタイムアウトで処理結果不明になり得る場合は、`processing` のまま放置せず、調査対象状態またはリカバリ方針を設計メモに残す。

### 次サイクル planner への入力

- `transfer_requests.status` の状態遷移表を追加するタスクを候補化する。
- 入力検証エラー、業務拒否、システム障害を区別した失敗記録方針を検討する。

## Finding 4: 競合更新時の残高保護方針がまだ実装可能な粒度で決まっていない

### 根拠

設計原則では振込をデータベーストランザクションで原子的に扱う方針がある。テスト戦略でも将来追加するテストとして並行実行時の残高競合テストが挙げられている。しかし、残高更新時に行ロック、条件付き UPDATE、分離レベル、楽観ロックなどのどれを使うかは未定義である。

### 影響

同じ口座から同時に複数の出金・振込が走ると、単純な「残高読取 → 残高不足判定 → 更新」では lost update や二重引落可能額の誤判定が起きる。学習用であっても、この部分が曖昧なまま実装されると、残高がマイナスにならないという基本原則を破る金融事故シナリオにつながる。

### 事故シナリオ

1. 口座 A の残高が 10,000 円。
2. 同時に 8,000 円の振込依頼が 2 件処理される。
3. 両方が更新前残高 10,000 円を読み、残高不足ではないと判定する。
4. 排他制御が不十分だと、2 件とも成功扱いになり、実質 16,000 円の出金を許してしまう。

### 推奨修正

初期実装では、PostgreSQL を前提に「対象口座行をトランザクション内でロックしてから残高判定・更新する」または「`balance_amount >= amount` を条件にした単一 UPDATE の更新件数で成功判定する」のどちらかを明示する。振込では振込元・振込先のロック順序を固定し、デッドロック回避方針も簡単に記録する。

### 次サイクル planner への入力

- 出金・振込時の残高競合制御方式を accepted scope 候補に入れる。
- 同一口座への並行出金・並行振込テストを MVP テスト候補に追加する。

## Finding 5: 監査ログと取引履歴のトランザクション境界が未整理

### 根拠

MVP と設計原則では、重要操作に監査ログを残し、すべての残高変更に取引履歴を残す方針がある。一方で、残高更新・取引履歴作成・振込依頼更新は同一トランザクションに含めると明記されているが、監査ログを同一トランザクションに含めるか、失敗時ログをどのように残すかはまだ未定義である。

### 影響

成功時の監査ログを業務トランザクション外で書くと、残高は変わったのに監査ログが欠落する可能性がある。逆に失敗時の監査ログを同一トランザクション内だけで書くと、ロールバックにより失敗証跡が消える可能性がある。金融事故調査では、成功取引だけでなく拒否・失敗・権限違反の追跡も重要である。

### 推奨修正

成功時の監査ログ、業務拒否時の監査ログ、システム障害時のログを分けて境界を決める。少なくとも、残高変更を伴う成功操作では「残高更新・取引履歴・振込依頼成功・成功監査ログ」の一貫性をどう担保するかを設計する。失敗監査ログは、業務データのロールバック後でも記録できる方針を別途検討する。

### 次サイクル planner への入力

- 監査ログのトランザクション境界を設計判断として docs に残すタスクを候補化する。
- 成功・業務拒否・システム障害の監査記録方針を分けて整理する。

## 人間確認事項

1. MVP の冪等性キーの一意スコープを `requested_by_user_id` 基準にするか、`source_account_id` 基準にするか。
2. 失敗した振込依頼をどの範囲まで `transfer_requests` に保存するか。特に権限不足や存在しない口座への依頼を保存対象に含めるか。
3. 取消・訂正を MVP で扱わない場合でも、将来のために `reversal` の意味と残高方向だけ先に定義するか。
4. 監査ログの失敗記録を業務 DB 内で扱うか、将来の外部ログ基盤を想定して分けるか。

## 総評

既存ドキュメントは、学習用ミニバンキングシステムとして必要な主要概念、MVP、セキュリティ、テスト観点を広く押さえている。特に、金額を整数で扱うこと、残高変更に取引履歴を残すこと、振込を原子的に扱うこと、冪等性キーで二重送金を防ぐことは明確である。

次の改善は、実装前に「曖昧だと金融事故につながる境界」を小さく具体化することが望ましい。優先度は、残高増減方向の定義、冪等性キー衝突時の扱い、並行更新時の残高保護、監査ログのトランザクション境界の順に高い。
