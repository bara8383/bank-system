## Finding 1: 監査ログの成功時整合性境界が未確定のまま残っている

- 重大度: Medium
- 観点: 監査証跡、ログ、権限境界、金融操作の否認防止
- 人間確認要否: 要

### 根拠

- `docs/design-principles.md` は重要操作に監査ログを残す原則を定義している。対象はログイン、顧客登録、口座作成、入金、出金、振込、権限変更で、実行者、対象、操作種別、実行日時、結果、失敗理由を残すとしている。
- 今回の差分では、入金、出金、振込成功時に同じデータベーストランザクションへ含める更新として、口座残高、取引履歴、振込依頼の成功状態更新が明示された。
- 一方で、この成功時境界に `audit_logs` を含めるか、業務 transaction commit 後に別経路で必ず残すか、監査ログ書き込み失敗時に業務処理を fail closed するかは確定していない。
- `docs/test-strategy.md` の追加テストも、残高と取引履歴の整合性を主対象としており、残高変更成功と監査ログ成功記録の整合性テストは追加されていない。
- implementer は `監査ログ書き込み失敗時の扱い` と `失敗時監査ログ` を未確認事項として残しているが、成功した残高変更の監査ログをどの境界で保証するかまでは分離できていない。

### 影響

将来の入金、出金、振込 API 実装時に、残高変更と取引履歴だけを transaction 内で確定し、監査ログを後続の best effort 処理として実装してしまう余地がある。その場合、残高変更は成功したが監査ログが欠落する事故が起き、誰がいつ実行した操作かを後から説明できなくなる。

攻撃または事故シナリオ:

- 権限を持つ内部ユーザーまたは侵害されたアカウントが出金・振込を実行する。
- 残高更新と取引履歴は commit されるが、監査ログ insert が失敗する、または実装から漏れる。
- 取引履歴から金額移動は見えるが、操作 actor、IP、User-Agent、成功/失敗結果が欠落し、不正操作や権限誤設定の調査が困難になる。

### 推奨修正

次サイクルでは、監査ログ専用の設計 scope として以下を docs に明記する。

- 残高変更を伴う成功操作では、監査ログ記録を必須とする。
- 成功監査ログを業務 transaction に含めるか、outbox など別方式で永続化を保証するかを決める。
- 監査ログ書き込み失敗時に業務処理を失敗させるか、補償・再送・運用アラートで扱うかを人間確認事項として判断する。
- 失敗監査ログは業務 transaction rollback と独立して残す必要があるかを決める。
- `docs/test-strategy.md` に、入金、出金、振込の成功・失敗時に監査ログが必ず作成され、秘密情報や過剰な個人情報を含まないことを確認するテスト観点を追加する。

## その他の観点

- 認証・認可: 今回差分では実装も方式確定もしていない。既存 docs の「未認証ユーザーは業務データへアクセス不可」「顧客は自分の口座のみ操作可」という原則との明確な矛盾はない。
- 入力検証: 金額は正の整数、MVP は JPY、残高は 0 未満にしないという追記は安全側。新たな入力検証リスクは見つけていない。
- 秘密情報: パスワード、トークン、秘密鍵を扱う実装差分はない。
- SQL injection: SQL、repository、DB schema の実装差分はないため、今回差分から直接の SQL injection リスクは発生していない。
- 権限境界: 管理者・運用担当者の権限表は未確定のまま。業務 API 実装前に別 scope で具体化が必要。

## 次サイクル planner への入力

- 監査ログ境界、失敗時扱い、閲覧権限、マスキング方針を次の security design scope 候補にする。
- accepted scope に含める場合は、成功監査ログ、失敗監査ログ、監査ログ書き込み失敗時の業務処理、閲覧可能ロール、マスキング対象、テスト観点を分けて扱う。
- 人間確認事項として、監査ログ書き込み失敗時の fail closed / 補償方式、失敗ログの rollback 独立性、`admin` と `operator` の閲覧・操作権限境界を残す。
- 認証方式、RBAC、入力検証、エラー応答、SQL injection 防止、ログマスキング標準は、業務 API または DB 実装前に別途 docs 化する。
