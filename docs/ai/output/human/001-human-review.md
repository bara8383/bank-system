# human-review

## plannerから依頼された人間最終確認項目について

### 1. MVPにおいてはrevearsalは実装に含めない
- 通常取引をまずクリアにするべき。
- reversal は「既存取引との関係」「二重取消防止」「取消可否」「監査」が増えて重い。

### 2. postgreSQLの行ロックで悲観ロック
- 学習のため。
- 将来的には用途によって分割する

### 3. 操作種別、送信元口座、ログインユーザー、リクエスト本文hash全部を含める
- 全部含めないと、冪等性として意味ないのでは？
- 何か拍子に、架空の送金などができてしまいそう。

### 4. MVPとしては拒否のみ
- 将来的には、既存結果を返却したい。
- 理由として、そちらの方がUXが良いと考える

### 5. MVPとしては失敗させる
- 将来的には別経路で保障させる
- 理由、UX向上のためやりたいが、MVPとしては重い

### 6. 独立して残す
- MVPとしては少し重いがこれは学習用としてやりたい。
- 監査ログは全部残す方針であるため。

### 7. 認証情報はcokkie だが、これだと、csrfに弱いのでcsrf対策用トークンも別に持つ。

### 8. adminとoperaterの違いがわかりません。
admin = システム管理者、operater = 銀行業務を代行する人と認識して、返答します。
adminは代行可能。operatorはMVPには含めない。
